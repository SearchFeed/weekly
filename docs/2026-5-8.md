# 前端&AI技术双周刊-2026.05.08


## 前端技术
- [AVA 8.0 发布：完全 ESM 化，新增条件测试修饰符](https://github.com/avajs/ava)
  <br>AVA 8.0 是这个流行的 Node.js 测试运行器的重要版本更新，完全转向 ESM。对于前端工程师，这个更新的价值在于：1) 完全 ESM 化，与现代 JavaScript 生态对齐，告别 CommonJS；2) 新增 test.skipIf() 和 test.runIf() 修饰符，可根据条件跳过或运行测试，灵活性更高；3) 简化测试配置，减少样板代码。AVA 一直以并发测试、简洁 API、清晰的错误信息著称，v8.0 的 ESM 化意味着更好的 Tree-shaking、更快的加载速度、更现代的开发体验。如果你正在寻找轻量级、高性能的测试框架，或者需要从 CommonJS 测试迁移到 ESM，AVA 8.0 是值得评估的选择。

- [Hono Node.js Adapter 2.0：性能大幅提升，比 Express 快 4.1 倍](https://github.com/honojs/node-server)
  <br>Hono Node.js Adapter 2.0 让原本为 Cloudflare Workers、Deno、Bun 设计的 Hono 框架能够在 Node.js 上运行，且性能优异。对于前端工程师，这个适配器的价值在于：1) 性能大幅提升，基准测试显示 Hono 在 Node.js 上达到 85,405 reqs/s，比 Express 的 20,803 reqs/s 快 4.1 倍；2) 代码跨运行时复用，同一份代码可在 Cloudflare Workers、Deno、Bun、Node.js 上运行；3) 完整功能支持，包括 WebSocket、静态文件服务、HTTPS、UNIX domain socket；4) 轻量级优化，自动清理未完成的请求，可选择覆盖全局 Request/Response 对象以提高性能。如果你正在评估 Express 的替代方案，或需要在 Node.js 环境中使用 Hono，这个适配器是高效的选择。

- [Fresh 2.3：Deno 全栈框架新增 WebSocket 与 View Transitions 支持](https://fresh.deno.dev/)
  <br>Fresh 是 Deno 原生的全栈 Web 框架，v2.3 带来了重要的功能更新。对于前端工程师，这个框架的价值在于：1) 零配置理念，无构建步骤、无 node_modules，只需一个文件即可启动带路由、JSX、islands 的服务器；2) 服务端渲染默认，页面零 JS 发送到浏览器（除非使用 islands 交互部分），首屏性能极佳；3) v2.3 新增一等的 WebSocket 支持，适合实时应用；4) View Transitions API 支持，只需一个配置项即可实现原生感的页面过渡；5) 表单、HTML 流式传输、OpenTelemetry、CSP 等企业级特性开箱即用。Fresh 已在生产环境验证（如巴西电商平台 Deco.cx，实现 5x 页面加载速度和 30% 转化率提升）。如果你正在寻找轻量级、高性能的全栈框架，特别是 Deno 技术栈，Fresh 值得尝试。

- [Rolldown 1.0 发布：基于 Rust 的高性能 JavaScript 打包器](https://rolldown.rs/)
  <br>Rolldown 1.0 是一个基于 Rust 的高性能 JavaScript 打包器，结合了 esbuild 的速度和 Rollup 的插件生态系统。对于前端工程师，这个工具的价值在于：1) 性能优异，在 19k 模块的基准测试中，Rolldown 仅需 1.61 秒，远快于 Rollup + esbuild 的 40.10 秒；2) Rollup API 兼容，可直接使用现有的 Rollup 插件生态；3) esbuild 功能对等，内置转换、define、inject、minify 等；4) 为 Vite 8+ 提供统一的打包器。随着现代 Node.js 原生支持 .ts 文件、ESM 和 JSON 导入，Rolldown 的核心价值在于发布阶段，提供最小的 tree-shaken 运行产物。如果你正在寻找比 Rollup 更快、比 esbuild 功能更全的打包方案，Rolldown 值得评估。

- [PM2 7.0 发布：重构代码库，减少依赖，支持 Bun 应用](https://pm2.keymetrics.io/)
  <br>PM2 7.0 是这个流行的 Node.js 进程管理器的重要更新。本次重构大幅减少了依赖占用，使工具更轻量化。更重要的是，PM2 7.0 将集群模式和监控代理扩展到了 Bun 应用，这意味着使用 Bun 作为运行时的项目也可以享受 PM2 的进程管理、负载均衡和监控能力。对于前端工程师，特别是维护生产环境 Node.js/Bun 应用的团队，这一更新值得评估：减少依赖意味着更快的安装速度和更小的攻击面；Bun 支持则为新技术栈提供了成熟的进程管理方案。

- [Electrobun 1.18：基于 Bun 的轻量级跨平台桌面应用框架](https://electrobun.dev/)
  <br>Electrobun 是一个基于 Bun 构建的跨平台桌面应用框架，旨在提供轻量级的桌面应用开发体验。v1.18 是最新版本。对于前端工程师，这个框架的价值在于：1) 相比 Electron，基于 Bun 的运行时可能带来更小的包体积和更快的启动速度；2) 适合需要轻量级桌面应用的场景，如工具类应用、系统托盘应用等；3) 使用现代 JavaScript/TypeScript 技术栈，与前端开发体验一致。如果你正在寻找 Electron 的轻量级替代方案，或需要构建小型桌面应用，可以关注 Electrobun 的发展。

- [opentype.js 1.3.5：浏览器端 OpenType/TrueType 字体解析与生成库](https://opentype.js.org/)
  <br>opentype.js 是一个在浏览器和 Node.js 环境中读写 OpenType 和 TrueType 字体的 JavaScript 库。v1.3.5 是即将发布的 2.0 版本的预览版。对于前端工程师，这个库的价值在于：1) 直接在浏览器中访问字体的字形数据，无需服务器处理；2) 支持 WOFF、OTF、TTF 格式，以及连字、字距调整、表情符号；3) 可从零创建字体，适用于自定义字体设计工具；4) 适合字体可视化、字体编辑器、自定义字体渲染等场景。如果你需要在 Web 应用中实现字体相关的创意功能或专业工具，这个库是很好的起点。

- [pnpm 11.0 发布：纯 ESM、默认供应链安全、SQLite 存储索引](https://pnpm.io/blog/releases/11.0)
  <br>pnpm 11.0 是一次重大版本更新，标志着这个流行的包管理器向更安全、更高效的方向演进。关键变化包括：1) 转为纯 ESM，要求 Node.js 22+；2) 默认开启供应链安全保护（新包需等待 1 天、阻止非标准子依赖）；3) 用 SQLite 数据库替换 JSON 文件索引，减少系统调用；4) 全局安装隔离，避免包间冲突；5) 原生发布流程，不再依赖 npm CLI。对于前端工程师，这意味着更安全的依赖管理、更快的安装速度，以及更清晰的配置分离。如果团队使用 monorepo 或对供应链安全有要求，这一版本值得重点评估。

- [Nano Stores 1.3：仅 286 字节的跨框架状态管理器](https://github.com/nanostores/nanostores)
  <br>Nano Stores 1.3 是一个极轻量级（仅 286 字节）的状态管理库，支持原子存储和派生存储，兼容所有主流框架（React、Vue、Svelte 等）以及原生 JavaScript。对于前端工程师来说，这种框架无关的设计和极小的体积意味着可以在不同项目中统一状态管理方案，避免因框架切换带来的重构成本。特别适合微前端架构、渐进式迁移、或需要同时维护多个框架版本的项目。其原子化设计也便于性能优化和状态隔离。

- [Safari Technology Preview 243 发布，包含多项 CSS 与无障碍修复](https://webkit.org/blog/17953/release-notes-for-safari-technology-preview-243/)
  <br>Safari Technology Preview 243 带来了大量 CSS 布局、动画、无障碍功能的修复与改进。对前端工程师而言，这期更新值得关注的有：CSS Containment Level 2 对 contain: style 的支持、text-autospace 的 insert 关键字、flex 与 grid 布局的多个 bug 修复、:has() 选择器性能优化、以及多项无障碍交互修复。这些变更反映了浏览器对现代 CSS 规范与无障碍标准的持续对齐，对于需要精确控制布局、性能或无障碍体验的项目，了解这些底层修复有助于避免兼容性问题并优化实现方案。

- [Node.js 26.1 引入实验性 node:ffi 模块](https://nodejs.org/en/blog/release/v26.1.0)
  <br>Node.js 26.1 正式引入了一个实验性的 node:ffi 模块，允许 JavaScript 代码直接调用原生动态库的符号。虽然被标记为“不安全”且需要开启相应标志，但这个 API 的出现意味着服务端 JavaScript 与原生 C/C++ 生态的互操作方式有了新的可能性，对于需要高性能计算、硬件访问或与已有原生库集成的场景（如高性能图像处理、特定硬件 SDK、特定系统库调用等），这是一个重要的技术基础设施。对于前端工程师，如果团队有全栈、工具链、性能关键模块等需求，值得关注这一能力未来的稳定化进程。

- [Node.js 26.0.0 发布](https://nodejs.org/en/blog/release/v26.0.0)
  <br>Node.js 26 是当前发布线的重要版本更新，默认启用了 Temporal API，并升级到 V8 14.6 与 Undici 8。对前端工程师来说，这条资讯的价值不只是“版本发布”，而是运行时能力边界的变化：Temporal 终于可以作为更现代的时间处理方案直接进入服务端 JavaScript 实践；V8 升级带来了新的语言能力与性能基础；Undici 更新则影响基于 Node 的 HTTP 请求链路。如果团队有 SSR、BFF、构建服务、Node 工具链或全栈项目，这一版值得尽快评估兼容性与升级收益。
